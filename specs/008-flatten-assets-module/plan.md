# Implementation Plan: Flatten Assets Module

**Branch**: `008-flatten-assets-module` | **Date**: 2026-03-20 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/008-flatten-assets-module/spec.md`

## Summary

Move `VEN/src/simulator/assets/` to a top-level `VEN/src/assets/` module. This is a pure
file-system and `mod`/`use` path rename — no logic, types, or trait implementations change.
The five asset files (`pv`, `battery`, `ev`, `heater`, `base_load`) and their shared `mod.rs`
move verbatim. `simulator/mod.rs` gains a re-export bridge so existing call sites continue
to compile. `AssetEntry`, `SimState`, and `GridMeter` remain in `simulator/mod.rs` (deferred
to a later restructure).

## Implementation Sequence

### Step 1 — Create `VEN/src/assets/` with moved files

Copy the six files from `simulator/assets/` into the new top-level `assets/` directory.
Update only the module declarations inside `assets/mod.rs` if they reference the old path.
(The five sub-module files contain no `mod` declarations that reference `simulator/assets`.)

Files to create (content = current `simulator/assets/` content verbatim):

```
VEN/src/assets/mod.rs
VEN/src/assets/pv.rs
VEN/src/assets/battery.rs
VEN/src/assets/ev.rs
VEN/src/assets/heater.rs
VEN/src/assets/base_load.rs
```

No content changes needed in the five asset files — their `use` statements already use
absolute crate paths (`use crate::common::…`, `use super::history_from_buffer`, etc.) or
relative paths that remain valid within the new location.

### Step 2 — Update `simulator/mod.rs`

Remove the `pub mod assets;` declaration and replace with targeted re-exports:

```rust
// Remove:
pub mod assets;

// Add:
pub use crate::assets::{
    AssetState, AssetCapabilities, TickEnvironment,
    EnergyState, TimeWindow, ControlKind, ControlDescriptor,
    history_from_buffer,
};
```

Any existing `use super::assets::…` or `use crate::simulator::assets::…` within
`simulator/mod.rs` itself must also be updated to `use crate::assets::…`.

### Step 3 — Declare `mod assets` in `main.rs`

Add the top-level module declaration to `main.rs` alongside the existing `mod simulator;`:

```rust
mod assets;
mod simulator;
// … other mods …
```

### Step 4 — Build and iterate

```bash
cd VEN && cargo build
```

The compiler will flag every remaining stale reference. Fix each one. Common locations:
- `controller/planner.rs` — may reference `AssetState` variants
- `controller/dispatcher.rs` — references `AssetEntry`, `AssetState`
- `entities/asset.rs` — may reference asset capability types

### Step 5 — Delete `simulator/assets/`

Once `cargo build` is clean, delete the old directory:

```bash
rm -rf VEN/src/simulator/assets/
```

Run `cargo build` again to confirm nothing broke.

### Step 6 — Run tests

```bash
cargo test --workspace
```

Then deploy to Pi4 and run BDD:

```bash
docker compose -f tests/docker-compose.test.yml run --build --rm test-runner
```

Expected: 895 steps, 0 failures.

## Risk Register

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| Stale `use` paths in controller/ files | Medium | Compiler flags all of them; fix iteratively |
| `super::` relative imports inside asset files break after move | Low | Asset files use `use super::history_from_buffer` (still valid — `super` is `assets/mod.rs`) |
| Re-export bridge masks a call site that uses the wrong type | Low | After tests pass, optionally remove re-export to confirm all direct call sites updated |
| Pi4 Docker layer cache serves stale binary | Medium | Always pass `--build` flag to `docker compose run` |
