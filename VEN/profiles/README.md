# VEN Profiles

Per-VEN simulator/planner configuration, mounted read-only into each
container at `/config/profile.yaml` (`PROFILE_PATH`).

| File | Used by |
|---|---|
| `ven-1.yaml` … `ven-3.yaml` | the static production trio (`VEN/docker-compose.yml`) |
| `test.yaml`, `no_pv_test.yaml` | the E2E test stack |
| `fleet-ven-*.yaml` | generated by `scripts/gen_fleet_profiles.py` — never edit by hand, regenerated on every `fleet.sh up` |

## Personas (WP4.5, Phase 4)

`bash fleet.sh up N --personas eco:0.4,comfort:0.4,commuter:0.2` assigns each
generated VEN one of three presets (seeded, reproducible; ratios normalized).
Presets are pure configuration on top of the Phase-4 features — the single
source of truth is `scripts/personas.py`:

| Persona | EV session mode | Comfort curve (asset `ev`) | Asset-mix nudges |
|---|---|---|---|
| `eco` | `OPPORTUNISTIC` — free energy only | aggressive: bids 0.10 → 0.03 €/kWh | EV always, battery 90 %, base load 0.2–0.4 kW |
| `comfort` | `ASAP` — cost-blind | flat, high: 0.45 → 0.40 €/kWh | EV 80 %, battery 50 %, base load 0.5–0.8 kW |
| `commuter` | `BY_DEADLINE`, ready 07:00 UTC, budget 2 € recorded | mid: 0.30 → 0.15 €/kWh | EV always, battery 30 %, base load 0.2–0.4 kW |

The persona lands in two places:

1. **Profile generation** (`gen_fleet_profiles.py --personas …`): asset-mix
   probabilities and base-load range; the persona name is written to
   `VEN/fleet/manifest.json`.
2. **Experiment runtime** (`experiments/run_experiment.py --personas`): before
   the scenario starts, each fleet VEN gets its persona's EV session
   (mode/target/departure/budget) and comfort-curve override; both are removed
   afterwards. `experiments/kpi.py --manifest VEN/fleet/manifest.json` then
   adds a per-persona KPI block to the report.

Note: the commuter's budget is recorded on the session; the hard MILP cost cap
binds only in `MAX_COST` mode (WP4.1-c) — for `BY_DEADLINE` it is documentation
of intent, not a constraint.
